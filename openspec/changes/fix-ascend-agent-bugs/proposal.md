## Why

Four reported production bugs in AscendAgent block core user flows: semantic memory never persists facts, image uploads fail with HTTP 500 (`Invalid mime type "file"`), PDF document summarization fails with HTTP 502 from Docling (`404 Not Found`), and the AscendMemory search call returns HTTP 500 because of a query-parameter naming mismatch (`userId` vs `user_id`) introduced by commit `94790bc`. A follow-up codebase audit surfaced ~12 additional latent issues across the memory, chat-history, RAG-retrieval, and ingestion paths — including a Soft-RAG threshold that is bypassed at query time, S3-key path traversal on document upload, no upload MIME allowlist, no multipart size limits, missing idempotency on the HTTP ingestion endpoint, declared-but-not-applied chat-history TTL, silent swallow of failed-fact inserts, missing `wipe`/`delete` client methods, and undocumented MinIO auto-ingestion flow (`app.ingestion.auto.enabled` is OFF by default; users currently must call `POST /api/ingestion/run` manually). All issues are localized to AscendAgent — external services in `docker-compose.yaml` are healthy.

## What Changes

- **Bug 1 — Semantic memory extraction**: harden `SemanticMemoryExtractor` so it reliably extracts a JSON array from thinking-model responses (e.g., MiniMax-M2.7) that prepend chain-of-thought before / instead of the JSON. Use `ChatResponseContentResolver` for content resolution, then a robust regex-based JSON-array extractor with balanced-bracket scanning. Tighten the extraction prompt to forbid prose / reasoning in the output. On parse failure, log the full raw response at WARN once but do not silently drop facts — also attempt a retry with a stricter "JSON only" instruction.
- **Bug 2 — Image upload MIME parsing**: in `ChatExecutor.handleImageContext`, guard against `MultipartFile.getContentType()` returning a value that is not a valid MIME type (missing `/`, `null`, or literally `"file"`). Fall back to `MimeTypeUtils.IMAGE_PNG` (or sniff the first bytes / filename extension) instead of throwing. Catch `InvalidMimeTypeException` explicitly so a malformed client header degrades gracefully rather than producing HTTP 500.
- **Bug 3 — Docling endpoint path**: fix the configured Docling API path. The current default `/v1/convert` returns 404 from `docling-serve` v0.x because the real upload endpoint is `/v1/convert/file` (multipart/form-data). Update `application.yaml` `app.docling.api-path` default and add a smoke test (or startup probe) so an obviously-wrong endpoint is detected at boot rather than at first user request.
- **Bug 4 — AscendMemory search query-param mismatch**: in `SemanticMemoryClient.performSearchCall` (line 47) the URI sends `userId={userId}`, but the AscendMemory FastAPI endpoint at `GET /api/v1/memory/search` expects `user_id` (snake_case, see `AscendMemory/src/api/rest/rest_endpoints.py:20-32`). FastAPI rejects the request as missing-required-param and returns HTTP 500. The insert call at line 87 already uses `user_id` correctly — only the search URI is stale. Fix is a one-line change to use `user_id={userId}`. Add an integration-style test that mocks AscendMemory and asserts the outgoing query string.
- **Bug 5 — Memory client gaps and silent failures** (audit findings A): add `wipeUserMemory(userId, embeddingProvider)` and `deleteMemory(...)` methods on `SemanticMemoryClient` to match the FastAPI endpoints in `AscendMemory/src/api/rest/rest_endpoints.py`. Fix `SemanticMemoryProperties` default `baseUrl` from `http://localhost:8770` to `http://localhost:7020` (matches docker-compose). Validate non-blank `userId` at the entry of `search`/`insert`/`wipe`/`delete` and short-circuit with a logged warning. In `SemanticMemoryExtractor`, replace the silent `forEach(memoryClient.insertMemory)` with an explicit success/failure tally that logs `Failed to insert {N}/{total} facts for user '{userId}'` when any insert fails.
- **Bug 6 — Chat-history TTL is declared but never applied**: `application.yaml` sets `app.memory.chat-history.ttl: 86400s` but `PersistentChatMemory` never invokes `redisTemplate.expire(...)`. Inject the TTL via `@Value`/`@ConfigurationProperties` and call `expire(key, ttl)` after every `rightPush`/`rightPushAll`. Verify Postgres `chat_history` rows are also bounded (or document the expected pruning strategy in `application.yaml`).
- **Bug 7 — RAG retrieval bypasses the configured similarity threshold**: `RagRetrievalService` builds the `SearchRequest` with `SearchRequest.SIMILARITY_THRESHOLD_ACCEPT_ALL` and then post-filters by `app.rag.similarity-threshold` only on the top hit. This drops the entire RAG context if hit #1 is below threshold even when hits #2–#5 are above it. Apply `ragProperties.getSimilarityThreshold()` directly to the `SearchRequest` and remove (or correct) the post-filter so all hits above threshold are injected.
- **Bug 8 — Ingestion upload security and limits**: `IngestionController` ingests `MultipartFile` without sanitizing `getOriginalFilename()` (S3 key path-traversal: `../../etc/passwd.txt`), without enforcing a MIME allowlist, and `application.yaml` defines no `spring.servlet.multipart.max-file-size` / `max-request-size`. Add filename sanitization (replace anything outside `[A-Za-z0-9._-]` with `_`, strip leading dots), add a configurable allowlist (default: `application/pdf`, `text/markdown`, `text/plain`, `image/png`, `image/jpeg`, `image/webp`, common DOCX/PPTX), and configure multipart limits (default 25 MB file / 30 MB request, override-able).
- **Bug 9 — HTTP-upload ingestion has no idempotency**: `DocumentIngestionService` (the `POST /api/ingestion/upload` path) does NOT call `documentService.removeOldDocuments(...)` before adding new chunks, so re-uploading the same file produces duplicate Qdrant points and inflated RAG context. The `ManualIngestionService` (MinIO scan) flow already does this via the metadata store. Reuse the same dedup mechanism in the HTTP flow keyed by sanitized filename + content hash.
- **Bug 10 — Vision capability not enforced and container-mode URL defaults**: `ChatExecutor.handleImageContext` attaches media regardless of whether the selected provider/model supports vision (e.g., a non-VL MiniMax model receives an image and either errors or silently ignores it). Add a small capability map (provider+model → supportsImages) and reject the request with HTTP 400 + clear message when the user attaches an image to a non-vision model. Separately, the `unstructured.base-url` default `http://localhost:9080` only works when AscendAgent runs on the host; document that the docker profile (`application-docker.yaml`) overrides this to `http://ascend-unstructured:9080` and verify both paths.
- **Bug 11 — Documentation and RAG UX gaps**: the README explains MinIO + folder routing for `obsidian/` and `documents/`, but does not clearly document (a) that `app.ingestion.auto.enabled` is `false` by default and the user must call `POST /api/ingestion/run` to index files dropped into MinIO, (b) how to switch on the auto-poller, (c) the embedding-dimension/collection coupling (`ascendai-768` vs `ascendai-1536`) and what to do when switching providers, (d) how RAG, semantic memory, and chat history differ. Add a short "RAG ingestion lifecycle" section to `AscendAgent/README.md` and a corresponding ADR in `AscendAgent/docs/architecture/decisions/` covering the auto-OFF-by-default decision.

## Capabilities

### New Capabilities
(none — these are bug fixes against existing behavior)

### Modified Capabilities
- `semantic-memory-extraction`: Memory extractor MUST return parsed facts when the underlying LLM is a thinking model (response contains reasoning + JSON array).
- `image-prompt-handling`: Image-bearing prompt requests MUST succeed when the multipart `image` part has a missing or malformed `Content-Type`; the agent MUST infer or default the MIME type rather than 500.
- `document-ingestion-docling`: Docling client MUST target the documented `docling-serve` upload endpoint and the default configuration MUST work against the `docker-compose.yaml` instance out of the box.
- `semantic-memory-client`: AscendAgent's HTTP client for AscendMemory MUST use the snake_case query parameter `user_id` for the `GET /api/v1/memory/search` endpoint, matching the FastAPI server contract; MUST expose `wipe`/`delete` operations; MUST validate `user_id` at entry; default `baseUrl` MUST match docker-compose.
- `chat-history-persistence`: Redis chat-history entries MUST honor the configured TTL.
- `rag-retrieval`: The configured similarity threshold MUST be applied at vector-search time so all hits above threshold are returned, not only when the top hit clears the bar.
- `ingestion-security`: Document/image upload endpoints MUST sanitize filenames before constructing storage keys, enforce a MIME-type allowlist, and respect a configurable multipart size limit.
- `ingestion-correctness`: HTTP-upload ingestion MUST deduplicate prior chunks for the same document key before adding new ones; image-bearing prompts MUST validate the provider/model supports vision.
- `rag-documentation`: README and architecture docs MUST clearly describe the MinIO → ingestion → Qdrant → RAG flow, the auto-vs-manual ingestion toggle, and embedding-dimension/collection coupling when switching providers.
